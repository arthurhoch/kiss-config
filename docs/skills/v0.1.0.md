# KissConfig AI Skill v0.1.0

This Markdown file is a versioned AI skill for using **KissConfig** in other Java projects. It is intentionally self-contained so an AI assistant can load this one document, add the Maven dependency, write consumer code, and avoid inventing APIs.

Use this skill for release **0.1.0**. If the repository source is on a later <code>-SNAPSHOT</code>, consumer documentation should still use <code>0.1.0</code> unless the user explicitly asks for a snapshot build.

## Library Summary

Tiny zero-dependency Java 17+ configuration loader for properties, explicit env files, system properties, environment variables, interpolation, secret-aware reporting, and record mapping.

## Maven Dependency

~~~xml
<dependency>
  <groupId>io.github.arthurhoch</groupId>
  <artifactId>kiss-config</artifactId>
  <version>0.1.0</version>
</dependency>
~~~

## AI Usage Rules

- Target Java 17 or newer.
- Prefer the public package rooted at <code>io.github.arthurhoch.kissconfig</code>.
- Do not invent convenience APIs. Use only the public members listed in this file or in generated Javadocs for the same release.
- Keep examples small and explicit, matching the KISS philosophy.
- Do not add extra frameworks unless the consuming project already uses them.
- Keep older skill files in place when a new release is documented.

## Quick Example

~~~java
public record AppConfig(ServerConfig server) {}
public record ServerConfig(String host, int port) {}

AppConfig config = KissConfig.load(AppConfig.class);
~~~

## How To Use The Library

- `KissConfig.load(Class<T>)` maps configuration into an immutable record using default options.
- `KissConfig.builder()` customizes property file, env file, profile, search order, merge strategy, missing explicit file behavior, and result shape.
- Use `mapTo(Class<T>)` when only the mapped value is needed. Use `load(Class<T>)` when diagnostics, loaded sources, effective map, or the report are needed.
- Records are the primary supported mapping target for v0.1.0. Prefer immutable records in consuming projects.
- Use annotations from `io.github.arthurhoch.kissconfig.annotations` to tune mapping and secret behavior.
- Do not import `io.github.arthurhoch.kissconfig.internal.*` in consumer projects. Those classes are implementation details.

## Behavioral Contract For v0.1.0

- Default options are `application.properties`, no env file, no profile, `SearchOrder.defaults()`, and `MergeStrategy.FILL_MISSING_ONLY`.
- Search order is read order. With `FILL_MISSING_ONLY`, earlier sources win. With `OVERRIDE_EXISTING`, later sources win. With `FAIL_ON_DUPLICATE`, repeated canonical keys fail.
- `SearchOrder.defaults()` reads classpath library defaults, application classpath, JAR directory, working directory, Java system properties, and environment variables.
- Env files are explicit. They are not loaded unless configured by `envFile(...)` or by a search order containing an env-file location.
- Profile names must match `[a-zA-Z0-9_-]+`. Profiles add profile file variants; `SearchOrder.production()` does not enable a profile by itself.
- Secret values are masked in reports and safe values. Use `@Secret` for explicit secret fields and avoid logging raw `ConfigValue.value()` for secret keys.
- Expected failures use `ConfigException` and typed subclasses for load, parse, mapping, interpolation, duplicate key, invalid profile, and missing property cases.

## Practical Examples

### External-first production config

~~~java
AppConfig config = KissConfig.builder()
    .profile("prod")
    .envFile(".env")
    .searchOrder(SearchOrder.production())
    .mergeStrategy(MergeStrategy.FILL_MISSING_ONLY)
    .mapTo(AppConfig.class);
~~~

### Annotated record

~~~java
public record DatabaseConfig(
    @ConfigName("db.url") String url,
    @Required String username,
    @Secret String password,
    @DefaultValue("10") int poolSize
) {}
~~~
